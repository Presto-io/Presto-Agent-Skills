# v1.5 School Presentation Stack Research

**Date:** 2026-05-31
**Milestone:** v1.5 School Presentation Skill

## Question

What is the least-warping implementation route for a Markdown-first school presentation skill that needs formulas, tables, charts, images, videos, readable typography, automatic overflow pagination, and offline static delivery?

## Findings

- **Marp** is the cleanest existing Markdown slide converter. It uses Markdown plus directives and CSS theming, and can export to HTML, PDF, and PowerPoint. This validates the idea that Markdown can drive slide output, but it still treats PPTX as an export target rather than the primary stable runtime. Source: https://marp.app/
- **Slidev** is a strong web-native slide authoring environment. Its export docs confirm PDF/PPTX/PNG/Markdown output, but PPTX export captures slides as images, so text is not selectable. This confirms that PPTX export is possible but lossy for editable semantics. Source: https://sli.dev/guide/exporting
- **reveal.js** directly supports Markdown slides through its built-in Markdown plugin, external Markdown files, slide separators, and speaker notes separators. This makes it a strong conceptual base for HTML-first output. Source: https://revealjs.com/markdown/
- **Pandoc** has mature PPTX generation with a `reference.pptx` model and predefined PowerPoint layout selection. This is useful inspiration for fixed layout routing, but it pulls the project back into the unstable PPTX-first path the user wants to avoid. Source: https://pandoc.org/MANUAL.html#powerpoint
- **Deckset** demonstrates a polished Markdown-to-presentation product surface and supports many content types relevant to this skill: tables, formulas, auto-scaling, images, videos, graphs, presenter notes, and custom theming. It is useful aesthetic evidence but not a portable runtime base. Source: https://docs.deckset.com/English.lproj/getting-started.html
- The bundled Codex **Presentations** skill is high quality for PPTX, but it is built around artifact-tool presentation JSX and editable PowerPoint output, not a persistent Markdown-to-HTML workflow. It can inform QA expectations such as contact-sheet review and anti-generic-design checks, but not the v1.5 core architecture.

## Decision

Use a Markdown logical-slide intermediate plus a custom school-style HTML renderer. Borrow:

- Marp's simple Markdown authoring posture.
- reveal.js's web-native slide runtime concepts.
- Pandoc's fixed-layout routing idea.
- Deckset's content support and visual polish expectations.
- The local Presentations skill's QA mindset, especially source reading, design-system locking, and avoiding generic AI-looking output.

Do not use PPTX as the v1.5 primary output.

## Build Implications

- Markdown should describe logical slides, not final physical pages.
- Renderer should map each logical slide to a fixed layout family.
- Renderer should auto-split overflow into continuation pages.
- Images should preserve aspect ratio and use contain-style placement by default.
- Charts should be rendered from structured data into SVG or self-contained HTML assets.
- Formulas should use a local/static KaTeX-style path or an equivalent bundled renderer.
- Videos need documented packaging behavior: small videos can be embedded; large videos should use an assets/zip fallback instead of making the HTML unusably large.
- Verification should include rendered previews or screenshots, not just file existence.
