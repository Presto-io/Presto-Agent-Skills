# Phase 08 Plan: School Presentation Skill

**Milestone:** v1.5 School Presentation Skill
**Created:** 2026-05-31
**Mode:** mvp

## Goal

Create the first `school-presentation` skill: a Markdown-first workflow that turns source materials into logical slides and renders a blue-green school-style offline HTML deck with fixed layout families, overflow-safe pagination, and deterministic verification.

## Scope

- Add `skills/school-presentation/SKILL.md` as the canonical skill source.
- Add skill-local school identity references and provenance notes under `skills/school-presentation/references/`.
- Add `skills/school-presentation/templates/school-presentation-full.md` as the reusable logical-slide Markdown template.
- Add `skills/school-presentation/scripts/school-presentation.sh` as the render entrypoint for example generation, HTML rendering, and verification helpers.
- Keep HTML as the primary v1.5 output; do not introduce PPTX, Keynote, hosted delivery, OCR, or annotation features.
- Enforce the 50 MB output cap and document the large-video fallback path.

## Tasks

1. Define the canonical skill contract.
   - Files: `skills/school-presentation/SKILL.md`
   - Action: Write the trigger wording, objective, inputs, process, outputs, verification, success criteria, safety boundaries, and runtime adapter notes for Codex, Claude Code, Gemini CLI, OpenCode, OpenClaw, and Hermes Agent.
   - Verify: The skill clearly says when to use it, how to normalize into Markdown first, and how to keep runtime differences out of the canonical workflow.
   - Acceptance: The skill is concise, portable, and points all rendering work at the Markdown logical-slide intermediate.

2. Lock the school identity reference layout.
   - Files: `skills/school-presentation/references/`
   - Action: Extract the school logo, slogan, palette, reusable decorative marks, and provenance notes from the provided PPTX/POTX materials into a skill-local reference tree such as `references/identity/`.
   - Verify: The extracted references are readable, skill-local, and clearly traceable back to the source material.
   - Acceptance: The skill owns its identity assets and provenance without needing repository-wide directory changes.

3. Design the logical-slide Markdown schema and reusable template.
   - Files: `skills/school-presentation/templates/school-presentation-full.md`
   - Action: Define a persistent Markdown structure where each logical slide is a reviewable source block with frontmatter, slide headings, layout intent, speaker notes, warnings, and content sections that can carry formulas, tables, charts, images, videos, and text.
   - Verify: The template remains human-editable, matches the shared Markdown normalization contract, and supports explicit layout override while defaulting to automatic layout selection.
   - Acceptance: A contributor can author one deck as stable Markdown without bypassing the source of truth.

4. Build the HTML rendering pipeline.
   - Files: `skills/school-presentation/scripts/school-presentation.sh`
   - Action: Implement a stable script entrypoint that can generate the example template, render Markdown into offline HTML, package assets, preserve image aspect ratios, split overflowing logical slides into multiple physical pages, and apply a small fixed layout library.
   - Verify: The same input renders repeatably, the generated deck stays under 50 MB or uses the documented large-media fallback, and the HTML remains usable offline when media size permits.
   - Acceptance: Rendering is deterministic, self-contained when possible, and explicit about oversized-video behavior.

5. Validate against a representative school deck.
   - Files: the new skill files plus generated verification artifacts
   - Action: Render a representative fixture deck that exercises formulas, tables, charts, images, videos, speaker notes, warnings, and overflow splitting; then compare repeated renders and inspect the preview artifacts.
   - Verify: The rendered outputs are stable across repeated runs, the preview artifacts show the expected content types, and the deck reads as school-branded rather than generic.
   - Acceptance: Human visual review remains the final gate, but the automated checks prove repeatability and template-control correctness.

## Verification

- Render the same Markdown input twice and compare the normalized HTML output or manifest for stability.
- Confirm the deck/package stays at or below 50 MB, or falls back to the documented oversized-media path without breaking offline usability.
- Confirm the sample verification artifacts include formulas, tables, charts, images, videos, readable typography, and overflow splitting.
- Confirm the template and skill documentation keep Markdown as the editable source of truth.

## Success Criteria

- `skills/school-presentation/SKILL.md` defines the trigger, workflow, outputs, safety boundaries, verification, and runtime adapter notes for all required runtimes.
- `skills/school-presentation/references/` contains extracted school identity assets or provenance notes from the provided PPTX/POTX materials.
- `skills/school-presentation/templates/school-presentation-full.md` provides a reusable Markdown logical-slide intermediate with fixed layout intents, formulas, tables, charts, images, videos, and speaker notes.
- `skills/school-presentation/scripts/school-presentation.sh` renders Markdown into an offline self-contained HTML presentation when feasible and documents the large-video fallback behavior.
- The renderer preserves image aspect ratios, applies fixed school-style layout families, and automatically splits overflowing logical slides into multiple physical HTML pages.
- The provided fixture produces preview and verification artifacts showing formulas, tables, charts, media, readable typography, overflow splitting, and non-generic school visual design.
