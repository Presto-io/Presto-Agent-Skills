<interfaces>
This is a documentation/template phase. There are no code interfaces, package installs, runtime harnesses, schemas, UI routes, APIs, databases, or deployment contracts.

The executor must preserve these decisions from `02-CONTEXT.md`:
- D-01: The primary product is a single portable `SKILL.md` format, not a multi-layer adapter system.
- D-02: The canonical `SKILL.md` should be directly installable/usable by supported agents wherever their skill-loading model permits it.
- D-03: Runtime-specific material should be limited to compatibility notes.
- D-04: Do not design extra adapter abstractions beyond what is necessary to document installation/use differences.
- D-05: If a runtime needs a different path or discovery mechanism, document it as installation/loading guidance while preserving one source skill file.
- D-06: Avoid duplicating skill logic into per-runtime files.
- D-07: `templates/skill/SKILL.md` is the central artifact contributors copy from.
- D-08: Adapter template and compatibility matrix updates must support the single-file story.
- D-09: Keep language simple, direct, and install-oriented.
- D-10: Codex, Claude Code, Gemini CLI, OpenCode, OpenClaw, and Hermes Agent are required v1 targets.
- D-11: OpenClaw and Hermes Agent are first-class targets.
- D-12: Where exact runtime behavior is not locally verified, say what must be checked during installation instead of inventing richer adapter design.
</interfaces>
</context>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Canonical skill template to runtime notes | Runtime-specific syntax can leak into portable workflow instructions. |
| Compatibility docs to real runtime behavior | Documentation can overclaim support for runtimes not locally executed. |
| Contributor path to adapter docs | Contributors can be forced into unnecessary multi-file work if docs imply adapters are mandatory. |
| Planning records to execution scope | GSD status updates can accidentally mark Phase 3 example work complete. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | `templates/skill/SKILL.md` | mitigate | Verify canonical template remains runtime-neutral and includes all required sections. |
| T-02-02 | Tampering | `templates/adapter/runtime-adapter.md`, `adapters/README.md` | mitigate | Reframe adapter docs as minimal compatibility notes and prevent duplicated skill logic. |
| T-02-03 | Information Disclosure | safety/tool sections | mitigate | Require safety boundaries for external commands, network, credentials, and file writes in templates. |
| T-02-04 | Spoofing | compatibility claims | mitigate | Mark unverified path/loading behavior as installation-time checks rather than verified support. |
| T-02-05 | Denial of Service | phase scope | mitigate | Keep scaffold CLI, schema/lint, runtime harnesses, generated wrappers, and publishing out of Phase 2. |
| T-02-06 | Elevation of Privilege | runtime tool permissions | mitigate | Adapter template must include tool/permission differences and verification checklist. |
</threat_model>
