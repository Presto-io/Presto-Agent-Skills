# Clean Delivery Directory Contract

本契约定义所有会创建文件的技能都必须遵守的最小交付目录规则。目标是让用户打开交付目录时，只看见当前最新版 Markdown 和当前最终交付产物，避免 agent 把中间文件、验证证据、失败产物和多轮修改副本散落在用户硬盘中。

## Required Layout

```text
<delivery-root>/
├── <project-name>.md
├── <project-name>.<delivery-format>
├── sources/
├── assets/
├── history/
│   └── 001/
└── .work/
```

根据任务实际需要，可以有多个最终交付产物，例如同级 `.pdf`、`.pptx`、`.html` 或 `.xlsx`。没有内容的目录不必提前创建。

## Root Allowlist

交付根目录一级只允许存放：

1. 当前最新版、用户可直接审阅的 Markdown。
2. 从该 Markdown 生成或与其组成同一交付版本的最终产物。
3. `sources/`、`assets/`、`history/` 和隐藏 `.work/` 目录。

根目录不得出现 manifest、status、model、JSON、日志、diff、截图、验证报告、缓存、临时文件、失败产物、staging 文件或带随意版本后缀的重复交付文件。

## Supporting Directories

- `sources/`：用户明确要求保留的原始资料。不要为了方便自动复制所有输入。
- `assets/`：当前 Markdown 或交付产物需要持续引用的图片、字体、音视频等资源。
- `history/`：已经成功交付、后来被新版本替换的完整旧版本。
- `.work/`：本次生成所需的临时文件。成功或失败收尾后都应清空；不需要时不创建。

技能确实需要其他长期资料目录时，必须在该技能的 artifact contract 中说明用途，并保持为有名称、有边界的目录；不得直接散落到交付根目录。

## Revision Rules

根目录中的稳定文件名永远代表当前最新版。禁止产生 `final-final`、`最终版2`、`最新版-修改后`、日期副本或其他平铺版本文件。

每次修改按以下顺序执行：

1. 在 `.work/<run-id>/` 中生成一整套候选 Markdown 和交付产物。
2. 只执行证明交付可用所必需的最小验证，不为验证大量创建持久文件。
3. 如果生成或验证失败，不修改根目录中的当前版本；删除本次不完整候选和临时证据。
4. 如果候选与当前版本完全相同，不创建重复历史版本。
5. 如果候选成功且内容发生变化，把根目录中的上一套 Markdown 和交付产物整体移动到下一个 `history/<sequence>/`。
6. 将候选版本作为一个整体发布到根目录，并保持稳定文件名。
7. 清空本次 `.work/<run-id>/`；`.work/` 为空时可以删除。

`history/` 使用零填充递增序号，例如 `001/`、`002/`。同一次成功交付产生的 Markdown 和所有最终产物必须进入同一个历史目录，不得分别编号。技能不得自动删除历史版本；如需压缩或删除，由用户另行决定。

## Verification And Failure

- 验证优先使用退出状态、内存检查、标准错误和系统临时目录。
- 仅当产物本身需要携带 manifest 或报告时，才把它列为最终交付产物。
- 成功后不得为了证明成功而在交付目录保留额外验证文件。
- 失败时不得留下空 PDF、损坏 PPTX、半成品 HTML 或看似可交付的文件。
- 不得删除或移动不在当前技能明确管理范围内的现有文件。
- 清理未知文件、旧 agent 产物或用户资料前，必须先列出计划并取得用户确认。

## Skill Authoring Requirement

所有会写文件的 `SKILL.md` 必须：

- 指向本契约，或声明更严格但不冲突的交付规则。
- 列出根目录允许出现的 Markdown 和最终交付产物。
- 说明 `sources/`、`assets/`、`history/`、`.work/` 中实际会使用的目录。
- 说明多轮修改时如何归档上一套成功版本。
- 说明失败时如何保持当前成功版本不变并清理半成品。

单技能可以定义自己的字段、文件名和渲染规则，但不能放宽根目录整洁、版本成套归档和失败不污染当前交付的要求。
