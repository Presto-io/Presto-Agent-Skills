# Agent 硬盘产物整理提示词

把下面提示词交给能够访问本地文件系统的 coding agent 使用。首次运行默认只审计和提出方案；只有用户确认后才执行移动、归档或删除。

```text
你现在要帮助我整理此前由 AI agent 在硬盘上产生的大量散乱文件。你的目标不是“尽可能多地删除”，而是在不损坏用户原始资料、项目源码、版本历史和当前可用交付物的前提下，把每个交付目录整理成干净、可理解、可回退的结构。

【工作范围】

只检查我明确提供的这些根目录：

- <在这里填写允许检查的目录 1>
- <在这里填写允许检查的目录 2>

不得越过这些目录，不得扫描我的整个主目录，不得访问凭据目录、浏览器数据、聊天记录、照片图库、邮件、云盘同步目录或其他未授权位置。遇到符号链接时先报告其真实目标；目标在授权范围外时不要跟随。

【最终目录原则】

每个实际交付目录的一级只保留：

1. 当前最新版、用户可直接阅读和编辑的 Markdown；
2. 与该 Markdown 对应的当前最终交付产物，例如 PDF、PPTX、HTML、DOCX、XLSX、Typst；
3. sources/：用户明确需要保留的原始资料；
4. assets/：当前 Markdown 或交付产物持续引用的图片、字体、音视频等资源；
5. history/：旧的完整成功版本，每一版独立成套归档；
6. 必要时存在的隐藏 .work/，但整理完成后必须为空并删除。

manifest、status、model、diagnostics、JSON、日志、diff、截图、缓存、临时文件、测试输出、staging、失败产物、空文件、重复副本和带 final-final、最终版2、修改后、最新版、副本等后缀的文件，不得继续平铺在交付目录一级。

【安全禁令】

- 第一阶段只读审计，不移动、不重命名、不覆盖、不删除任何文件。
- 不运行会修改项目状态的构建、渲染、格式化、包管理或 Git 命令。
- 不修改 .git、.planning、node_modules、虚拟环境、系统目录或应用数据目录。
- 不把“看起来像垃圾”当作删除依据。无法确定用途的文件一律标为“待确认”。
- 不删除唯一副本，不覆盖同名文件，不打破 Markdown、HTML、PPTX、Typst 对相对资源的引用。
- 不把用户原始材料误判为 agent 中间产物。
- 不创建大量扫描报告、哈希清单、截图或验证文件。审计过程尽量在内存和系统临时目录完成，最终只生成一份总报告。

【第一阶段：只读审计】

先识别每个候选交付目录，并按以下类别盘点文件：

A. 当前候选 Markdown；
B. 当前候选最终交付产物；
C. 可能的旧成功版本；
D. 用户原始资料；
E. 当前产物依赖的 assets；
F. agent 中间模型、manifest、status、日志、diff、验证截图、缓存、staging 和临时文件；
G. 生成失败或明显不完整的产物；
H. 无法判断用途的文件。

判断“当前版本”时综合使用文件内容、修改时间、文件名、成套关系、非空状态和引用关系，不能只看文件名中的 final 或最新版。对于 PDF、PPTX、DOCX、XLSX、HTML 等产物，只做非破坏性检查，例如是否存在、是否非空、容器能否读取、是否与相邻 Markdown 名称和时间合理对应。不要为了验证而重新生成一套文件。

对疑似重复文件，可计算哈希，但哈希结果只用于当前判断，不要在每个目录保存 sidecar。完全相同的文件标记为“字节重复”；内容不同但名称相似的文件标记为“版本候选”，不得擅自删除。

检查 Markdown、HTML、Typst 等文本文件中的相对资源引用。任何被当前版本引用的图片、字体、音视频或附件都必须保留，并计划移动到 assets/ 时同步修正引用。二进制文档中资源关系无法可靠判断时，保守保留并标记待确认。

完成审计后，只输出一份清理方案报告，至少包含：

1. 授权扫描范围；
2. 发现的候选交付目录；
3. 每个目录建议保留在一级的当前 Markdown 和最终产物；
4. 建议移入 sources/ 的文件；
5. 建议移入 assets/ 的文件及需要同步修改的引用；
6. 建议按成套版本移入 history/001、history/002 的文件；
7. 建议删除的明确临时或失败文件，并逐项说明依据；
8. 无法判断、必须由我确认的文件；
9. 预计执行的移动、重命名、删除数量；
10. 风险和可回退方案。

报告中把计划操作分成“安全移动”“需要确认的归档”“需要明确批准的删除”三组。然后停下来等待我的确认，不得直接执行。

【第二阶段：确认后执行】

只有在我明确回复批准某一版方案后，才能执行被批准的操作。执行前再次确认目标目录仍与审计时一致；如果文件发生变化，停止并重新报告，不要在过期方案上继续。

执行时遵守以下顺序：

1. 创建确实需要的 sources/、assets/、history/ 目录；空目录不要创建。
2. 先整理旧成功版本。一次成功版本的 Markdown 和所有对应交付产物必须整体进入同一个 history/<三位递增序号>/，不能拆散。
3. 再整理 sources/ 和 assets/。移动 assets 后同步修正文本文件中的相对路径，并做最小只读检查。
4. 最后整理一级目录，只保留当前 Markdown、当前交付产物和规定目录。
5. 删除操作只针对我明确批准的文件。优先移动到本次集中隔离目录，而不是立即永久删除；隔离目录的位置必须由我确认。
6. 不处理未批准项，不顺手扩大范围。

当前文件使用稳定名称。禁止制造 final-final、最终版2、最新版-修改后、日期副本等新的平铺版本名。如果需要保留旧版本，使用 history/001、history/002 成套归档；根目录永远表示当前最新版。

【执行后的验证与收尾】

执行完成后，只做必要检查：当前 Markdown 和最终产物存在且非空；文本资源引用没有因移动而失效；每个 history 版本保持成套；一级目录没有遗留 manifest、日志、缓存、失败产物或重复版本。

不要为了证明整理成功而创建新的 manifest、status JSON、截图集、diff 集或逐目录报告。最终只更新最初那一份总报告，记录实际执行的移动、重命名、隔离和删除操作，以及仍待用户决定的项目。删除临时工作目录，并明确说明哪些文件完全没有触碰。

如果任何一步存在不确定性，宁可保留并询问，也不要猜测。你的成功标准是：用户一眼能看懂当前交付是什么，旧版本可以成套回退，原始资料和依赖资源有固定位置，程序垃圾不再散落，同时没有误删任何有价值的文件。
```
